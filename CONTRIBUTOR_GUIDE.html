3 The Method — how we build (this is the important part)

This is exactly how the whole thing got built. Do it this way and you can't go wrong.

1
Pick ONE thing you want.
A plugin, an effect, a feature — something you love in Pro Tools / After Effects / wherever. One thing. Not ten.
2
Screenshot EVERYTHING about it.
This is the secret. Take snapshots of exactly what you want — the layout, the knobs, the menus, what it looks like, how it behaves. Pictures beat words. Hand the AI the visuals.
3
Describe it in obsessive detail.
Don't be vague. Say exactly what it does, how it should look, how it should feel. The more of what's in your head you get out, the closer it builds to what you actually want.
4
Research FIRST, build second.
For anything real, tell the AI: "research how this works first, then build it." It'll send a swarm of agents out, get the real facts, and build on truth instead of guessing. Don't let it wing the big stuff.
5
Look at it. React. Refine.
Build it, open it in the browser, look hard. Love it? Keep it. Off? Tell it exactly what's wrong and run it back. That's the whole loop.
⚡ When to flip on ULTRACODEFor big jobs — a whole new feature, a complex plugin, the menu system, anything that needs deep research — turn on ultracode in Claude Code. It makes the AI go exhaustive and use the research swarms automatically. For tiny tweaks, normal mode is fine. Big = ultracode. Small = normal.

4 The Rules — so it's order, not chaos

A few simple rules keep us from breaking each other's work.

→
One thing at a time, and say what you're on. Tell the crew what you're building so two people don't build the same thing.
→
Don't break what works. Run it in the browser and make sure it works before you push. If it's broken, don't ship it.
→
Big changes go on a branch, not straight to main. Make your own branch, build there, then open a Pull Request so it can get a look before it lands. Small fixes are fine direct.
→
Read the plan docs first. HANDOFF.md + DEMARTIN_DAW_PLAN.md. Build on what's there, don't reinvent it.
→
Commit with a clear message. Say what you changed in one line, so everyone can follow the history.
🎛️ Real talk on audio pluginsThe basic browser audio blocks sound like garbage — that's a known ceiling, not your fault. To make plugins that actually sound pro, we use real DSP compiled to WebAssembly, and the best tool for that is Faust (there are already .dsp files in the vst/ folder — that's the road). Want to build a great-sounding plugin? Have your AI research the Faust → WebAssembly path first. The AI can write the Faust code — it just needs the right pipeline.
