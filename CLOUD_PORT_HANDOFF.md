## The ONE architecture decision
**AI calls fire straight from the browser, not through our server.** This is non-negotiable because:
- It's the only way **localhost LM Studio** works (the browser is on the user's machine, so `localhost` = THEIR machine; a hosted server's `localhost` = the wrong machine).
- Our server never touches their key or their audio → no liability, no key storage, no bandwidth bill.

### Provider CORS reality (from web research)
- **Browser-direct, no proxy:** LM Studio/Ollama (user flips ON "Enable CORS"), **Anthropic** (needs header `anthropic-dangerous-direct-browser-access: true`), **OpenRouter** (native CORS + it's a gateway to OpenAI/Groq/Gemini/Claude with one key).
- **Blocked from browser → need a thin proxy OR route via OpenRouter:** OpenAI, Groq, Google Gemini. (`dangerouslyAllowBrowser` does NOT fix this — common trap.)
- **Forced server jobs:** web-research scraping (`/api/research`), and heavy video/MP3 encode (or push to WASM on the user's CPU).
- **localhost from HTTPS:** allowed in Chrome/Edge/Firefox (loopback = secure-context exception); expect a one-time Chrome "local network" prompt. **Safari can't reach local HTTP at all → Safari users = cloud key only.**

## Feature portability
**✅ Crosses free/easy:** Kitchen (everything, already serverless) · Studio core + WAV bounce (already Web Audio/OfflineAudioContext; `WEB_DEMO` mode exists) · editor compositing (`drawFrameInto`) · chat/persona/memory · builds/agents/plugins/projects JSON → IndexedDB · key-set checks · PDF/DOCX via pdf.js+JSZip.

**🔧 Crosses with work:** every AI "talk to the plugin" call (Vocal Doctor, beatbrain, plugin-macros, image-prompt polish) → repoint at browser-direct key (work per call-site, ~4–5 rooms) · MP3 320 export → lamejs/ffmpeg.wasm · native file/folder pickers + disk session → File System Access API (Chrome/Edge; download/upload fallback) · all persistence/autosave → **IndexedDB** (NOT localStorage — Studio projects embed base64 audio, too big for ~5MB) · editor media import/transcode/peaks → drag-drop + ffmpeg.wasm/WebCodecs (heavy) · editor MP4 export → WebCodecs + mp4-muxer (hardest single port).

**⛔ Can't be pure-browser — honest call:** host user's installed VST3/AU/Waves (scan/params/render) → **desktop-only**, label the button (our own DSP plugins still work) · native pro time-stretch → drop silently (in-browser WSOLA already plays) · local-GPU image gen → cloud substitute (Atlas BYO-key) · hardware/GPU-free checks + local model scan → drop · desktop self-updater → drop (site reloads) · DPAPI key storage → localStorage/WebCrypto (user's own key on user's own machine).

## Cost
Thin proxy (only web-research + optionally OpenAI/Groq/Gemini) = stateless forwarder, holds no keys, ~$0–5/mo (Cloudflare Workers / small Render/Fly). Use OpenRouter → proxy nothing for chat. Real money only appears with server-side video encode → keep encode on the user's CPU (WASM/WebCodecs).

## Storage with no accounts
Per-browser. localStorage for small (keys/prefs), **IndexedDB for big** (projects w/ base64 audio, beats, plugins, agents, gallery). Honest limit: tied to one browser/device, clear-cache wipes it, no cross-device sync. Offer "Export project to a file" as manual backup/transfer. Cross-device sync/accounts = the day you add a real backend (the stated vision needs none).
